# mdalchemy Requirements

## Purpose

mdalchemy converts Markdown files into beautiful, human-readable documents. The first supported output is standalone HTML, but the project should not be designed as "a Markdown-to-HTML string converter." It should be a document processing tool with a parser, document model, renderer layer, theme system, and CLI.

The motivating use case is the large amount of Markdown generated by LLMs and developer tools. That content is often useful but visually plain. mdalchemy should turn it into something people want to read, share, print, or archive.

## Source Specifications

Markdown is not governed by one single modern canonical specification. The original Markdown syntax guide is intentionally small and leaves ambiguities. CommonMark exists to remove those ambiguities and provide conformance examples.

Initial project decision:

- Core Markdown target: [CommonMark 0.31.2](https://spec.commonmark.org/0.31.2/).
- Historical reference: [Daring Fireball Markdown syntax](https://daringfireball.net/projects/markdown/syntax).
- Later extension target: [GitHub Flavored Markdown 0.29-gfm](https://github.github.com/gfm/).

When this plan says "full Markdown support," it means full CommonMark 0.31.2 support unless explicitly marked as an extension.

## Product Goals

1. Provide a CLI that converts an input `.md` file to an output document.
2. Produce visually polished HTML that reads like a finished article, report, note, or reference page.
3. Implement a custom Markdown parser in TypeScript.
4. Keep the parser and document model independent from HTML output.
5. Support built-in themes.
6. Support user-defined themes through config files.
7. Support nested sections, nested lists, blockquotes, code, links, images, raw HTML behavior, escapes, references, and all CommonMark block and inline constructs.
8. Expose enough internal structure that future renderers can target PDF or other formats.
9. Keep dependencies minimal and intentional.
10. Make the project easy to learn from, modify, and extend.

## Non-Goals For Version 1

- Real-time browser preview.
- A graphical editor.
- WYSIWYG Markdown editing.
- Full GitHub-compatible rendering in the first implementation.
- Plugin execution from arbitrary user code.
- Remote asset downloading by default.
- Server mode.
- Incremental parsing for live editing.
- Compatibility with every existing Markdown dialect.
- Perfect print-ready PDF output in the first release.

## Target Users

### Primary Users

- Developers who want to turn `README.md`, notes, design docs, and LLM outputs into clean HTML.
- Writers who keep drafts in Markdown but want attractive documents without building a website.
- People using LLMs who want generated Markdown to become readable reports.

### Secondary Users

- Tool builders who want a modular Markdown parser and renderer pipeline.
- Learners who want to understand how Markdown parsing, ASTs, rendering, and CLI tooling fit together.

## User Stories

1. As a user, I can run `mdalchemy input.md -o output.html` and get a standalone HTML file.
2. As a user, I can choose a built-in theme with `--theme`.
3. As a user, I can point to a custom theme config file.
4. As a user, I can render nested headings into a semantic document outline.
5. As a user, I can render nested lists and blockquotes accurately.
6. As a user, I can render fenced code blocks with language classes.
7. As a user, I can render links and images safely.
8. As a user, I can generate an output file next to the input file when `-o` is omitted.
9. As a user, I can inspect parsing or rendering warnings in the terminal.
10. As a user, I can run a conformance test suite to see how close the custom parser is to CommonMark.
11. As a maintainer, I can add a new renderer without changing parser internals.
12. As a maintainer, I can add a new built-in theme without changing parser or renderer logic.
13. As a maintainer, I can add Markdown extensions without contaminating the CommonMark core behavior.

## Functional Requirements

### CLI

- Must expose a bin command named `mdalchemy`.
- Must accept one input file path.
- Must accept an output file path with `--output` or `-o`.
- Must infer HTML output when the output extension is `.html`.
- Must default to HTML output for version 1.
- Must return non-zero exit codes for invalid input, unsupported output format, invalid config, and write failures.
- Must print useful errors without stack traces unless a debug flag is enabled.
- Must support `--theme <name-or-path>`.
- Must support `--config <path>`.
- Must support `--stdout` for piping rendered output.
- Must support `--strict` to fail on warnings that would otherwise be recoverable.
- Should support `--watch` after the first stable CLI release.

### Input Handling

- Must read UTF-8 Markdown files.
- Must preserve source positions in the AST for diagnostics.
- Must handle LF, CRLF, and CR line endings according to CommonMark.
- Must support files with or without trailing newlines.
- Must reject directories as file inputs.
- Must reject binary input with a clear error.
- Should expose an internal API that accepts strings directly, not just file paths.

### Markdown Parsing

- Must implement CommonMark block parsing.
- Must implement CommonMark inline parsing.
- Must produce a typed AST.
- Must keep source ranges on AST nodes.
- Must support nested container blocks.
- Must support nested sections as a derived structural feature from heading hierarchy.
- Must support link reference definitions across the document according to CommonMark precedence rules.
- Must preserve enough information for renderers to emit semantic output.
- Must not use an external Markdown parsing package.

### Document Model

- Must include a renderer-neutral document AST.
- Must distinguish block nodes from inline nodes.
- Must include a document root.
- Must include metadata for parse options, source file, diagnostics, and references.
- Should include a derived outline tree for headings.
- Should support future renderer-specific transformations without mutating parse results.

### Rendering

- Must render HTML as the first output format.
- Must use escaped output by default for text, attributes, URLs, and code.
- Must allow raw HTML behavior to follow CommonMark, with a safe-mode option available.
- Must generate a complete standalone HTML document by default.
- Must support rendering a fragment when requested by API or future CLI flag.
- Must include theme CSS inline by default for portability.
- Should support linked external CSS after v1 if requested.

### Theming

- Must ship with built-in themes.
- Must support user-defined themes from config files.
- Must separate theme tokens from renderer structure.
- Must allow typography, colors, spacing, code block styling, table styling, callout-like blockquote styling, and page width to be configured.
- Must validate user theme configs.
- Must provide helpful errors for invalid theme files.
- Should make themes portable and serializable.
- Should avoid requiring user-authored JavaScript for themes.

### Configuration

- Must support project config files.
- Must support explicit config path.
- Must have clear precedence rules.
- Must expose schema-like validation.
- Must support JSON first.
- Should support TOML or YAML later only if dependency policy allows it.

### Testing

- Must include unit tests for scanner, block parser, inline parser, renderer, theme validation, config loading, and CLI behavior.
- Must import or reference CommonMark spec examples for conformance testing.
- Must include snapshot tests for HTML renderer output.
- Must include fixture tests for complete Markdown documents.
- Must include regression tests for parser edge cases discovered during development.

## Markdown Feature Requirements

### CommonMark Core Blocks

- Thematic breaks.
- ATX headings.
- Setext headings.
- Indented code blocks.
- Fenced code blocks.
- HTML blocks.
- Link reference definitions.
- Paragraphs.
- Blank lines.
- Block quotes.
- List items.
- Ordered lists.
- Bullet lists.
- Tight lists.
- Loose lists.
- Nested lists.
- Mixed container nesting.

### CommonMark Core Inlines

- Backslash escapes.
- Entity references.
- Numeric character references.
- Code spans.
- Emphasis.
- Strong emphasis.
- Links.
- Images.
- Autolinks.
- Raw HTML.
- Hard line breaks.
- Soft line breaks.
- Textual content.

### Derived Features

Derived features are not separate Markdown syntax. They are computed from the parsed AST.

- Document title from first level-one heading, optional config override, or file basename.
- Table of contents from heading outline.
- Section wrappers for headings and following content.
- Heading anchors.
- Reading-friendly code block captions if future syntax or config provides them.
- Document metadata from config or the `frontmatter` extension.

### Extension Candidates

These are explicitly outside the CommonMark core unless added behind an extension flag.

- GitHub Flavored Markdown tables. Implemented as `gfm-table`.
- Task list items. Implemented as `gfm-task-list`.
- Strikethrough. Implemented as `gfm-strikethrough`.
- Autolink literal extension. Implemented as `gfm-literal-autolink`.
- Disallowed raw HTML filtering.
- Footnotes. Implemented as `gfm-footnote`.
- Definition lists.
- Frontmatter. Implemented as `frontmatter`.
- Admonitions.
- Math blocks and inline math.
- Mermaid diagrams.
- Additional syntax highlighting engines or language packs.

## Quality Requirements

### Correctness

- Parser behavior should be measured against CommonMark 0.31.2 examples.
- All parser decisions that differ from CommonMark must be explicitly documented as bugs or extension behavior.
- HTML renderer must not lose content.
- HTML renderer must avoid invalid nesting where the document model can express valid HTML.

### Security

- Text and attributes must be escaped.
- URLs must be normalized and validated before insertion into attributes.
- Raw HTML must be configurable:
  - `allow`: render according to CommonMark.
  - `escape`: escape raw HTML as text.
  - `strip`: remove raw HTML nodes.
- `javascript:` links must not be emitted in safe mode.
- Theme files must not execute code.
- The CLI must not fetch remote resources by default.

### Performance

- Target linear or near-linear parsing for typical documents.
- Avoid repeated whole-document string copies.
- Avoid recursion that can overflow on deeply nested input.
- Streaming is not required for v1, but the parser should be structured so large-file support is possible.
- Initial performance target: render a 1 MB Markdown file in under 2 seconds on a modern laptop after cold startup.

### Maintainability

- Keep modules small and named by responsibility.
- Use TypeScript discriminated unions for AST nodes.
- Keep parser phases explicit.
- Avoid global mutable parser state except within parser instances.
- Prefer pure functions for transforms and renderers where practical.
- Maintain fixtures that document expected behavior.

### Accessibility

- Generated HTML should use semantic elements.
- Themes must maintain contrast targets.
- Heading order should be preserved.
- Code blocks should be readable without color alone.
- Images should use alt text from Markdown image labels.
- Table of contents links should be keyboard navigable.

### Portability

- Output HTML should be standalone by default.
- Themes should not depend on remote fonts or external CDNs by default.
- CLI should run on macOS, Linux, and Windows.
- Path handling must use Node path utilities, not hard-coded separators.

## Configuration Requirements

Minimum viable `mdalchemy.config.json`:

```json
{
  "output": {
    "format": "html",
    "standalone": true
  },
  "theme": "serif",
  "markdown": {
    "profile": "commonmark",
    "extensions": []
  },
  "html": {
    "rawHtml": "allow",
    "headingAnchors": true,
    "tableOfContents": true
  }
}
```

Minimum viable custom theme file:

```json
{
  "name": "warm-report",
  "extends": "serif",
  "tokens": {
    "color.background": "#fffaf3",
    "color.text": "#241f1a",
    "color.accent": "#7a3f18",
    "font.body": "Georgia, serif",
    "font.mono": "ui-monospace, SFMono-Regular, Menlo, monospace",
    "layout.maxWidth": "760px",
    "space.block": "1.1rem"
  }
}
```

## Acceptance Criteria

The project is ready for 1.0 when:

- `mdalchemy input.md -o output.html` works for realistic Markdown documents.
- The parser and fragment HTML renderer pass the CommonMark 0.31.2 conformance suite in `commonmark` mode.
- All CommonMark feature categories are represented in tests.
- The renderer can produce standalone HTML with no remote dependencies.
- At least three built-in themes exist.
- A user can create a custom theme from documented tokens.
- Diagnostics are understandable.
- The codebase is modular enough that a second renderer can be prototyped without touching parser internals.
- Documentation explains architecture, parser design, renderer design, config, and contribution workflow.
